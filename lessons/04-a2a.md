# 4. A2A — agent calls agent

## Concept

MCP connects an agent to *tools*. A2A (Agent2Agent) connects an agent to
*other agents* — a different layer, easy to conflate with MCP since both
"add a capability" to an agent, but the thing on the other end is a full
agent (with its own LLM, its own tools) rather than a function.

## Analogy

Think of A2A as the difference between calling a library function (MCP tool)
and delegating a task to a coworker on another team (A2A). The coworker has
their own judgment, their own toolbox — you're not calling their internals,
you're handing them a task and trusting them to figure it out, then reporting
back.

## How it works

1. **Expose**: `to_a2a(root_agent, port=N)` wraps an agent in an ASGI app that
   publishes an **agent card** — a JSON manifest of what it can do — at
   `/.well-known/agent-card.json`. Serve it with uvicorn like any ASGI app.
2. **Consume**: `RemoteA2aAgent(agent_card=URL + AGENT_CARD_WELL_KNOWN_PATH)`
   is constructed from a URL, not a local Python object, then dropped into
   `sub_agents=[...]` exactly like a normal local agent would be.
3. The coordinator agent delegates via the **same** `transfer_to_agent`
   mechanism it would use for a local sub-agent — the LLM decides to hand off,
   ADK routes the call. The only difference is *where* that call actually
   goes: this time it crosses a process boundary as JSON-RPC over HTTP,
   possibly to a different machine, runs remotely, and returns.
4. The composition stacks: our agent card lists its **MCP tools** as A2A
   **skills** — `current_time`/`roll_dice` show up in the manifest even
   though they're MCP tools one layer down. MCP tool → agent → A2A skill.
5. This whole feature is flagged `[EXPERIMENTAL]` in ADK today — the warnings
   you'll see are expected and harmless.

## Look at

1. `adk_agent/a2a_server.py` — three lines, but know what they do: wraps
   `root_agent`, auto-generates the card, publishes it.
2. `a2a_consumer/agent.py` — `RemoteA2aAgent` built from a URL, then used as
   a normal `sub_agent` in a coordinator `LlmAgent`. Note it does its own
   independent credential loading (it's a standalone package, not importing
   from `adk_agent/`).

## Run this

Two terminals:

```bash
# terminal 1
make a2a-serve

# terminal 2
make a2a-card
```
Read the card JSON — confirm `current_time`/`roll_dice` are listed under
`skills` even though they're MCP tools underneath.

Then talk to the **consumer**, not the server:
```bash
uv run adk web
```
Pick `a2a_consumer` in the dropdown, ask "roll two dice." In the **Events**
trace, look for a `transfer_to_agent` event, then the call landing in
`utility_agent` — same shape as any local sub-agent delegation, just crossing
a process boundary.

## You'll know it clicked when

You can explain what happens if you ask `a2a_consumer` to roll dice while
`make a2a-serve` (terminal 1) isn't running — and specifically *when* it
fails: at consumer startup, or only at the moment it tries to delegate?
(The agent card is fetched once at startup — so also worth asking: what
happens if the remote agent's tools change *after* that?)

One more thing worth knowing before lesson 5c: `transfer_to_agent` (this
lesson) isn't the *only* way to reach a `RemoteA2aAgent` — it's one of two
independent choices (which agent, local-or-remote; and how you invoke it,
delegate-or-call-as-tool). Lesson 5c wraps this exact same remote agent in an
`AgentTool` instead, with very different control-flow consequences.
